** How would you measure the model's performance?

- We need a metric to evaluate the model's performance, also called a
  loss or cost function.

- For regression problems, we use RMSE (*Root Mean Squared Error*):
  \begin{equation}
  \text{RMSE}(\mathbf{X}, h) = \sqrt{\frac{1}{m}\sum_{i=1}^{m}\left(h(\mathbf{x}^{(i)}) - y^{(i)}\right)^2}
  \end{equation}

  #+attr_html: :width 400px :float nil:
  [[../img/rmse.png]]

  Where:
  + m ("sample") is the number of instances (rows/observations),
    i = 1..m
  + x^{(i)} ("features") is the i-th vector of all feature values
    except the label (if there are 10 features, then its length is 10)
  + h(x^{(i)}) ("hypothesis") is the model prediction for instance i
    (vector of length 10)
  + y^{(i)} ("target") is the actual label for instance i
  + *X* is a matrix with all feature values (except labels) of all
    instances: One row per instance, the i-th row is equal to the
    transpose of x^{(i)}, x^{(i)T}

- RSME code example:
  #+begin_src python :results output :exports both :session *Python* :python python3

- Why use RMSE?
  #+begin_quote
  The code example shows it: RMSE is in the same units as the target
  variable, which makes it more easily interpretable.
  #+end_quote

- What is common to all performance measures?
  #+begin_quote
  All performance measures are functions of the feature matrix *X* and
  the model (hypothesis h): they aggregate the difference between
  predictions h(x^{(i)}) and actual labels y^{(i)}) into a single
  scalar. The choice between them (RMSE, MAE, etc.) depends on how you
  want to weight errors — RMSE penalizes outliers more heavily (quadratic),
  MAE treats all errors equally (linear absolute value).
  #+end_quote

- Does the geometric argument for Lasso vs Ridge (from "[[https://campus.datacamp.com/courses/supervised-learning-with-scikit-learn/regression-55bf0d67-0cc2-4212-a6e5-bda0d07d3411?ex=11][Regularized
  regression]]" in DataCamp) also apply to MAE vs RMSE?
  #+begin_quote
  Yes — the same L1/L2 (measure) geometry applies. MAE uses absolute
  values (L1, diamond constraint region with corners on the axes),
  RMSE uses squared errors (L2, circular constraint region). L1 is
  more robust to outliers because large errors are not amplified by
  squaring. Lasso/MAE share the diamond geometry; Ridge/RMSE share the
  circle geometry.
  #+end_quote

* Create a test set
** Why?

- We need to evaluate the model on data it has never seen to avoid
  different forms of *bias*, which would mean that the data set does not
  represent the population - and then the performance measureis fake:
  the model looks as if it generalizes well but actually it does not.

- The test set must be set aside *before* any exploration (*no data
  snooping*): looking at the test set biases your intuitions and
  model choices.

** Random sampling

- The simplest approach: pick 20% of instances at random:
  #+begin_src python :results output :exports both :session *Python* :python python3
    from sklearn.model_selection import train_test_split

    train_set, test_set = train_test_split(
        housing_full, test_size=0.2, random_state=42)
    print(len(train_set), len(test_set))
  #+end_src

  #+RESULTS:
  : 16512 4128

- Problem: with a small dataset, random sampling may introduce
  *sampling bias* — the test set may not represent the full population.

- DataCamp's ~train_test_split~ examples avoided this issue by using
  clean, balanced datasets. Real-world data is rarely that
  cooperative.

- Worse: if the script is re-run with a different ~random_state~, a
  different 20% is selected each time. Over multiple runs the model
  indirectly sees the *whole dataset* — defeating the purpose of a test
  set entirely.

- Solution: fix ~random_state~ (we use 42 by convention) or save the
  test set to disk after the first split and never regenerate it.

- Problem with both solutions: if the dataset is refreshed with new
  instances, the fixed seed selects a different 20% and the saved test
  set is stale. The robust solution is to assign each instance a *hash*
  of a stable identifier (e.g. row index or district ID) and always
  put the same instances in the test set regardless of dataset order
  or size — new instances get assigned by their own hash, old ones
  stay where they were.

- The hash is essentially a *unique key* per row: the same input always
  produces the same hash value, so the train/test assignment is
  deterministic and stable across dataset updates.

- However, why is pure random sampling - even with stable hashing
  still potentially problematic for the housing dataset specifically?
  #+begin_quote
  Both the test and the training data set should be representative of
  the full dataset (a sample of the real population of CA districts).

  Domain experts say that the income is highly correlated with the
  house value - if the test set undersamples high or low income
  districts randomy, the error estimates will be systematically off.
  #+end_quote

** Clean the data

- Most ML algorithms cannot work with missing features. Options:
  1. Get rid of the corresponding districts[fn:2]
     #+begin_example python
     housing.dropna(subset=["total_bedrooms"])
     #+end_example
  2. Get rid of the whole attribute
     #+begin_example python
     housing.drop("total_bedrooms",axis=1)
     #+end_example
  3. Imputation: Set missing values to some value (zero, median etc.)
     #+begin_example python
     median = housing["total_bedrooms"].median()
     housing["total_bedrooms"] = housing["total_bedrooms"].fillna(median)
     #+end_example

- Which do you think is best?
  #+begin_quote
  Option 3, because the ~median~ is robust to outliers, all rows and all
  other features are preserved. Option 1 loses data, option 2 loses a
  potentially  useful predictor.
  #+end_quote

- A common solution is called "one-hot encoding": create one binary
  attribute per category. Only one category will be 1 (hot), all
  others (also called "dummies") will be 0 (cold).

** Feature scaling and transformation

- Algorithms that compute distances across features - regression, knn,
  neural networks etc. - care about scales: a feature with range
  0-40,000 will dominate a feature with range 0-15 independent of
  predictive importance.

- Is this relevant in our case? How would you find out? Compare for
  example the spread of values for =total_rooms= and =median_income=:

- *Answer:*
  #+begin_quote
  The =total_rooms= spread is in [2,39320] while the =median_income=
  spread is contained in (0,15]. The scales are vastly different.
  #+end_quote

- Common ways to scale all values to the same scale, are:

  1) *Min-max scaling* ("normalization"): shift all values, for all
     attributes, so that they range from 0 to 1. Subtract the min from
     all values (left shift), then divide by difference between min
     and max (normalize to [0,1]). It is strongly affected by outliers

  2) *Standardization* ("z-score normalization"): transform to a
     distribution with mean = 0 and a standard deviation = 1), without
     restricting the range. Subtract the mean (~mean~ becomes 0), then
     divide result by the standard deviation (~stddev~ becomes 1).

- Should scalers be fit to all the data, the training data, or the
  test data?
  #+begin_quote
  Scalers (~fit()~ or ~fit_transform~) should only be fitted to the
  training data (avoid data leakage from the test set). Once you have
  a trained scaler, you can use it to ~transform()~ any other set.
  #+end_quote

** Standardization (z-score normalization)

- Standardization is much less affected by outliers. For example, if a
  district has a median income equal to 100 (by mistake), instead of
  0-15. Min-max scaling maps it to 1 - all other values will be put
  into the 0-0.15 interval (and become irrelevant).

  For standardization, the outlier shifts the mean slightly, but one
  bad value does not move the statistics - the z-scores of all the
  other values remain almost unchanged. It is more concerned about the
  overall spread.

- *Bucketizing* the feature is another approach: You cut the
  distribution into roughly equal-sized buckets, and replace the
  feature value with the bucket index (similar to stratified
  sampling). Then you get a uniform-style distribution and no further
  scaling is needed.

- For *multimodal* distributions (like =housing_median_age=), you can
  treat the bucket IDs as categories (then they must be encoded
  numerically). Or you can add a feature for each of the modes,
  representing the similarity between the housing median age and that
  particular mode (e.g. houses aged 35 years).  This similarity is
  then measured using a radial basis function (RBF) that depends only
  on the distance between the input value and a fixed point, like the
  Gaussian RBF.

- For example, between housing age x and 35, it would be
  exp(-\gamma(x-35)^{2}). The hyperparameter \gamma determins how quickly the
  similarity measure decays as x moves away from 35.

** Custom similarity

Also defining =ClusterSimilarity= here: this code uses KMeans clustering
to identify the main clusters in the training data, and then uses a
Gaussian RBF kernel to measure how similar each district is to each
cluster's center — producing one new feature per cluster (Cluster 0
similarity, Cluster 1 similarity, etc.), where a higher value means
the district is closer to that cluster's centroid.

#+begin_src python :results none :session *Python* :python python3
  from sklearn.base import BaseEstimator, TransformerMixin
  from sklearn.metrics.pairwise import rbf_kernel
  from sklearn.cluster import KMeans

  class ClusterSimilarity(BaseEstimator, TransformerMixin):
    def __init__(self, n_clusters=10, gamma=1.0, random_state=None):
      self.n_clusters = n_clusters
      self.gamma = gamma
      self.random_state = random_state

    def fit(self, X, y=None, sample_weight=None):
      self.kmeans_ = KMeans(self.n_clusters, random_state=self.random_state)
      self.kmeans_.fit(X, sample_weight=sample_weight)
        return self  # always return self!

    def transform(self, X):
        return rbf_kernel(X, self.kmeans_.cluster_centers_, gamma=self.gamma)

    def get_feature_names_out(self, names=None):
        return [f"Cluster {i} similarity" for i in range(self.n_clusters)]
#+end_src

** Final transformer

- Recap of what the pipeline will do and why:
  + Impute missing values in numerical features by replacing them with
    the median.
  + Impute missing values in categorical features by the more frequent
    category.
  + One-hot-encode the categorical feature to make it numeric.
  + Compute and add additional ratio features: =bedrooms_ratio=,
    =rooms_per_house=, =people_per_house= that correlate with
    =median_house_value=.
  + Add cluster similarity features (more useful than latitude and
    longitude).
  + Replace features with a long tail by their logarithm (make them
    more symmetric).
  + Standardize all numerical features.
  + Exclude =housing_median_age= from all transformations.

- The following code achieves all this and is also modularized (using
  reusable functions):
  #+name: final_transform
  #+begin_src python :results none :session *Python* :python python3
    from sklearn.preprocessing import FunctionTransformer
    from sklearn.compose import make_column_selector

    def column_ratio(X):
      return X[:,[0]]/X[:,[1]]

    def ratio_name(function_transformer, feature_names_in):
      return ["ratio"] # feature names out

    def ratio_pipeline():
        return make_pipeline(
          SimpleImputer(strategy="median"),
          FunctionTransformer(column_ratio,feature_names_out=ratio_name),
          StandardScaler())

    log_pipeline = make_pipeline(
      SimpleImputer(strategy="median"),
      FunctionTransformer(np.log,feature_names_out="one-to-one"),
      StandardScaler())

    cluster_simil = ClusterSimilarity(n_clusters=10,gamma=1.,random_state=42)

    default_num_pipeline = make_pipeline(SimpleImputer(strategy="median"),
                                         StandardScaler())

    preprocessing = ColumnTransformer([
      ("bedrooms",ratio_pipeline(),["total_bedrooms","total_rooms"]),
      ("rooms_per_house",ratio_pipeline(),["total_rooms","households"]),
      ("people_per_house",ratio_pipeline(),["population","households"]),
      ("log",log_pipeline,["total_bedrooms","total_rooms","population",
                           "households","median_income"]),
      ("geo",cluster_simil,["latitude","longitude"]),
      ("cat",cat_pipeline,make_column_selector(dtype_include=object)),
    ],remainder=default_num_pipeline) # remaining: housing_median_age
  #+end_src

- Why is =housing_median_age= excluded from all of these extra
  transformations (except simple imputation and standard scaling)?
  #+begin_quote
  The other features are special-cased because of specific problems
  they have: ratios (bedrooms/rooms, rooms/households,
  population/households) are used because the raw counts alone aren't
  very informative — it's the ratio that carries the signal. The log
  transform is used for heavily skewed features like
  ~population~. Cluster similarity is used for ~latitude~/~longitude~
  because raw coordinates aren't linearly related to price — proximity
  to specific clusters is what matters.

  ~housing_median_age~ has none of these issues: it's not a raw count
  that only makes sense as a ratio, it isn't heavily skewed, and it
  isn't geographic. It's already a well-behaved numeric feature, so

** Three questions: trust in algorithms / new data: what to do / AI writes to binary
*** Question 1: Where do these algorithms come from, and why should I trust them?

This was Justin's question at the start of the show.

Answer: don't trust blindly, go deep(er) if you can.

The algorithms we use (linear regression, decision trees, k-means,
etc.) come from scikit-learn — though you could always code them from
scratch if you distrusted the library (Joel Grus's book /Data Science
from Scratch/ takes that approach, for the curious/masochistic). This
is also a good way to learn applied Python instead of wading through
200 pages of basic programming instruction (that's what I did!).

Strictly speaking, the only way to fully trust a library function is
to code it yourself. Short of that, trust in scikit-learn rests on:
- It ships with its own testing suite
- It is used by a huge number of people, including many with the skill and
  motivation to read the implementation code, not just call the interface
- It is fully open source — anyone diligent (or suspicious) enough can read
  the source and find mistakes. And mistakes do happen: the codebase runs to
  millions of lines, and researchers have built reputations by spotting
  errors in results that "didn't look right" and tracing them back to bugs
  in the library

scikit-learn is the most popular general-purpose ML library, but not
the only one — many smaller, specialized libraries exist and can
outperform scikit-learn on their narrow focus precisely because they
don't try to do everything.

- scikit-learn source code: https://github.com/scikit-learn/scikit-learn

*** Question 2: When I get new data after training/testing a model, do I have to retrain it?

This was Tori's question at the end of the show.

Answer: It depends on what kind of "new data" you mean.

**** New observations (same features, same target)

No, you generally do not retrain from scratch every time new rows come
in.  Real-time systems like Netflix recommendations or the transformer
models behind ChatGPT do not retrain overnight on every new
observation — that would be enormously expensive.

Adding observations is a statistics question, not an architecture
question: more rows almost always improves statistical significance (a
larger, more representative sample), but that is not the same as
practical significance.  If your model is already good enough for its
purpose, more data may not be worth the retraining cost. Whether to
retrain is a judgment call based on the situation, not an automatic
requirement.

That said, "no rush" is not "never" — if the real-world relationship
between your features and target shifts over time (tastes change,
markets shift), a model trained on old data will eventually degrade
even though nothing in the pipeline itself changed. This is the
"concept drift" the linked article (below) describes, and it's the
reason production models get periodically retrained even without
adding new features.

**** New features

This is a fundamentally harder problem. Adding a feature means going
back to the beginning of the pipeline: is it categorical or numerical?
How should it be encoded or scaled? This changes the architecture of
your model, not just its training data — there is no way to bolt a new
feature onto an already-trained model. The whole pipeline (imputer,
encoder, scaler, model) has to be rebuilt and refit.

Reference: on when and why production models degrade and need
retraining (the more general "concept drift" framing beyond what we
covered in class):
https://www.evidentlyai.com/ml-in-production/concept-drift

*** Question 3: Elon Musk claimed AI would soon write all code straight into binary, with no need for human-readable code. What's your view, and what does this mean for CS study?

I'm skeptical, for a specific reason: human-readable source code
exists so that we — humans — can trust what the machine is
doing. Cutting out the source code and going straight to binary would
remove the only layer we have for verifying, auditing, and debugging
what the system produces. That's not a minor convenience; it's close
to removing the mechanism for trust itself.  This has been done
already, but only in a small number of very specialized cases — not as
a general practice.

It's also worth weighing this claim against Musk's track record on
technology timelines: he has been saying since 2016 that fully
autonomous cars were imminent, and they still are not fully there
today. I would not bet on this binary-code claim playing out on his
stated timeline either.

What this means for CS study: the more grounded version of this trend
— AI increasingly writing and assisting with code — does have real
implications.  It shifts emphasis away from syntax memorization and
toward understanding what correct behavior looks like, system design,
critical reading and debugging of generated code, and the theoretical
foundations (algorithms, data structures, complexity) that don't
change regardless of who writes the implementation.

** How do you know which model to pick?

It's a good question. The honest answer would be: you don't know it in
advance, strictly speaking. You can pick a model based on experience
with a domain (the data, or other models that worked), or based on
visual insights (your sample data look as if a linear function would
approximate them), but you don't know - which is why we don't commit
to one in advance!

** Why linear regression first?

It's a cheap baseline: No hyperparameters to tune, trains instantly,
is fully interpretable (you get linear coefficients and you can read
the effect of each feature straight off them), tells you something
about the data (linear growth of features). If it does not work, it's
a signal that the relationships are more complex (interactions,
non-linearities), which is exactly why you might move to something
like decision trees (which accounts for non-linear relationships).

** What's the actual model selection process then?

1. Quickly try a handful of different model families (linear,
   tree-based, ensemble - like in the textbook example)
2. Compare them with cross-validation (not the test set!)
3. Shortlist promising candidates.
4. Fine-tuning hyperparameters.

The right model is picked empirically. There are also a few
mathematical methods.

The most relevant of these is the "no free lunch theorem". It's a
formal, provable result ⁠[[https://www.cs.ubc.ca/~hutter/earg/papers07/00585893.pdf][(Wolpert & Macready, 1997)]], which says that no
single model is universally best across all possible datasets,
averaged over all possible data-generating processes. So there is no
analytical shortcut to "the right model".

I bet this is discussed somewhere in the textbook, too, I just haven't
found it yet.

- Why are we not using the ~.score()~ method here, which returns the R^{2}
  coefficient instead of RMSE?
  #+begin_quote
  1. RMSE comes out in dollars.
  2. RMSE squares the error before averaging (reduces the impact of
     wildly bad predictions).
  3. R^{2} is defined relative to how much the target varies in *this*
     dataset. That makes it awkward to compare across different data
     slices or datasets — a model can look "good" (high R²) simply
     because the target has high natural variance, even if its
     dollar-error is large. RMSE doesn't have that confound.
  4. Sticking to RMSE is consistent to what we've done (an absolute
     metric, instead of a relative one). Relative metrics are harder
     to interpret.
  #+end_quote

- The model seems to be *underfitting* the data: the housing values
  range between $120k and $265k and a typical error of $68k is
  unsatisfying - the model does not "get" the existing granularity.

- The countermeasures:
  1) Fit a more powerful (for this data) model
  2) Feed training algorithm with better features
  3) Reduce the constraints on the model = less regularization - lower
     alpha = smaller penalty for higher order coefficients; add
     polynomial features - makes the model adapt more flexibly.

* Fine-tune the system

- If you have a shortlist of promising models you need to fine-tune
  your model, i.e. fiddle with the hyperparameters - settings you
  choose before training (alpha for regression, maximum tree depth for
  decision trees, and number of trees in the forest for random
  forest).

- Grid Search Cross-Validation will do the search for the best
  hyperparameter values for you. The Randomized Search CV is better
  when the hyperparameter search space is large.

- Another way to fine-tune the system (of models) is to combine the
  best-performing models.

- *With all these model combinations, does this not make ML very
  intransparent - almost a black box?*

  #+begin_quote
  The pipeline itself stays transparent: every transformation -
  ratios, log transforms, cluster similarity, scaling, one-hot
  encoding - is explicit, named, and inspectable. You can always
  answer "what happened to this raw column?" precisely.

  The model itself is where opacity creeps in, and it scales with
  model complexity. ~LinearRegression~ is fully transparent - each
  coefficient tells you exactly how much a feature moves the
  prediction.  ~DecisionTreeRegressor~ is still traceable - even a
  fairly deep tree lets you follow the exact chain of splits behind
  one prediction.  ~RandomForestRegressor~ is where it really becomes a
  black box: a prediction is the average of many trees, each seeing
  different bootstrapped data and feature subsets, so there's no
  single "reason" for a prediction anymore.

  This is a genuine tradeoff, not a side effect: more flexible models
  fit more of the real pattern in the data, but their internal logic
  becomes harder for a human to audit directly. That's why tools like
  ~feature_importances_~, permutation importance, and partial dependence
  plots exist for random forests - they attempt to open the box back
  up after the fact, even though you can't read off a clean equation
  the way you can with linear regression. The more powerful the model,
  the more trust has to come from validation (cross-validation, test
  error, importance analysis) rather than from being able to just read
  the model.
  #+end_quote
